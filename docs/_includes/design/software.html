Software & Autonomy

The final stack has three main software regions: Jetson host-native ORB-SLAM2 CUDA, a Dockerized ROS 2 Jazzy workspace, and ESP-IDF firmware with micro-ROS. The ROS side owns TF, odometry fusion, Klann geometry evaluation, controller/gait nodes, slam_toolbox, Nav2, RViz visualization, and the browser operator interface.

ROS 2 packages
  • dsy_motor_msgs: compact bridge messages for six-motor RPM commands and raw encoder/timing odometry.
  • hexapod_control_interfaces: leg phase, motor state, actuation state, and body-state messages.
  • hexapod_hardware_cpp: motor-state aggregation, raw encoder unwrap, Klann geometry/LUT tools, body-state estimation, weighted odometry fusion, startup pose, WT901 parsing, LCD/status nodes, and visualization markers.
  • hexapod_nav_cpp: MPPI/local-planner-oriented phase prediction and command generation, including final phase wrapping and forward-direction mirror/sign corrections.
  • hexapod_orbslam_udp_bridge: UDP receiver for ORB-SLAM2 pose, map points, and depth-derived virtual scans, plus slam_toolbox launch/configuration.
  • hexapod_operator_interface: browser GUI for map viewing, teleop velocities, waypoint goals, map save/load, e-stop state, and diagnostic status.

Frame and mapping ownership

TF ownership is explicit to prevent conflicting transforms. slam_toolbox owns map → odom. The weighted fusion node owns odom → base_link. The ORB-SLAM UDP bridge publishes /hexapod/orbslam_odom, /orbslam/map_points, and /scan, but does not publish a competing odom → base_link transform in the fused launch path. The camera/scan frame is mounted 11 in, or 0.2794 m, forward of base_link.

The final ORB/scan launch defaults use scale calibration, receive-time stamping by default, scan decimation for Jetson load, and no second ROS-axis conversion when the host ORB-SLAM process already emits ROS axes.
