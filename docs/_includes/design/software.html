Software and Autonomy Design

The software design separates perception/localization, proprioceptive odometry, planning/control, and low-level actuation so that any one subsystem can be tested independently. This section describes why the stack is partitioned this way; the Implementation page lists the concrete ROS 2 packages and firmware behavior.

Jetson kinematic linkage model

The runtime robot model on the Jetson is a measured Klann-linkage observer, not only a display asset. Raw 17-bit single-turn encoder counts arrive from the ESP32/micro-ROS bridge, are unwrapped across motor rotations, divided by the 50:1 gearbox ratio, and converted into output crank phase for each of the six legs.

For each crank phase, the host evaluates the measured linkage geometry as a precomputed lookup table. The table stores foot position, foot velocity, and a stance-confidence estimate over the crank cycle. During odometry, legs with high stance confidence are treated as approximate no-slip contacts; their predicted foot velocities are combined in a least-squares body-twist solve to estimate planar chassis motion.

This makes the Jetson model a fallback and fusion component for autonomy. ORB-SLAM2 provides camera-based motion and sparse structure when tracking is good, while the linkage model gives the planner and map stack a proprioceptive motion estimate when visual tracking is weak, delayed, or temporarily unavailable. Calibration terms such as sign/mirror corrections and body-twist scale factors are kept explicit because fabrication tolerances, foot slip, and ground compliance affect the physical robot.

Frame and estimator ownership

The design keeps planar mapping deterministic: slam_toolbox owns map → odom, the weighted fusion node owns odom → base_link, and the ORB-SLAM bridge publishes odometry, map points, and scan data without also publishing a competing body transform. Roll and pitch are intentionally kept out of the 2D mapping transform.

The autonomy path is intentionally operator-supervised. The robot can run mapping/localization continuously, accept teleop velocities or Nav2 goals from the browser interface, and save/load maps so that a stage environment can be prepared before a performance rather than reconstructed live under time pressure.
